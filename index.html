Personal Project

2-Player Interactive Chess

Java — Full Rule Enforcement with Swing GUI

I built this mostly to see if I could. A fully working chess game in Java, no libraries, just figuring it out from scratch. Chess sounds simple until you actually try to model the insanely complex logic of it in code.

The game has a full GUI built with Java Swing. Two players take turns clicking to move pieces on a rendered board. I implemented the full ruleset: every piece has its own legal move generation, and the game handles check, checkmate, castling, en passant, and pawn promotion. Getting check detection right was the hardest part. Every candidate move has to be simulated first to make sure it doesn't leave your own king exposed, which adds a layer of complexity to basically every move in the game.

The bigger challenge was keeping the codebase organized. Each piece is its own class, the board state is managed separately from the rendering, and the move validation logic sits in between. It got complex fast, and a lot of what I learned from this project was less about chess and more about structuring code that actually scales.

Full rule enforcement: check, checkmate, castling, en passant, and pawn promotion
Graphical board built with Java Swing, fully interactive click-based interface
Each piece is its own class with individual move logic, board state managed centrally
Move validation simulates each candidate move before allowing it

Personal Project

Super Mario Bros. Platform Tiler

Python — Built from Scratch, No Game Libraries

I built a Super Mario Bros.-inspired platformer in Python to see how games actually work under the hood. Things like the main loop, how input gets processed, how everything stays in sync each frame.

The levels are tile-based, so the world is basically a grid of platforms, gaps, and walls that the player moves through. I wrote the movement, gravity, and collision detection myself. Getting the collision to feel right took more iteration than I expected. There are a lot of edge cases when a character is landing on a tile at an angle or clipping a corner.

It was a good first look at what it means to write a program that runs continuously rather than just executing top to bottom. Different way of thinking about code structure.

Tile-based level system with grid-defined world layout
Basic player physics: gravity, movement, and jumping
Tile collision detection so the player interacts correctly with platforms and walls

ECE 1100 — Discovery Project

Light-Reactive Lamp

For my ECE 1100 Discovery Project, I built an ambient lamp that automatically adjusts its brightness based on the surrounding light level. When the room is dark, the lamp turns on. When there is enough ambient light, it turns off. The goal was to explore embedded systems, sensor integration, and basic electronics — all from scratch, with no prior hardware experience.

🎯

Goals & Objectives

  • Learn microcontroller programming using the ESP32-C6
  • Understand sensor integration with an LM393 light sensor module
  • Control a WS2812B 16-LED RGB ring using the Adafruit NeoPixel library
  • Build a complete working circuit from a breadboard prototype
🛠

Skills Obtained

  • Arduino-framework C++ on the ESP32-C6 (RISC-V core)
  • Breadboard wiring, GPIO pin mapping, and circuit debugging
  • Serial monitor debugging and iterative hardware troubleshooting
  • WS2812B LED control with the Adafruit NeoPixel library
  • Flashing firmware via Arduino IDE with correct board configuration
✅

Successes

  • Successfully flashed the ESP32-C6 after resolving board framework issues
  • Got the WS2812B ring responding to light sensor readings
  • Debugged wiring issues systematically using Serial Monitor output
  • Identified and worked around GPIO pin conflicts specific to the ESP32-C6
⚠️

Failures & Lessons

  • PlatformIO lacked Arduino framework support for the ESP32-C6 — switched to Arduino IDE
  • 3-pin LM393 module only outputs a digital signal, limiting smooth dimming
  • Light sensitivity was difficult to tune — the LM393 uses a hard digital threshold, so it struggled to detect gradual room darkening and could not reliably tell when the room became dark vs. slightly dim
  • Multiple GPIO pins conflicted with NeoPixel signal timing on the C6
  • Loose wire connections caused hours of unexplained failures
  • Hardware debugging requires patience and methodical isolation

Final Code

#include <Adafruit_NeoPixel.h>

const int ldrPin  = 5;   // LM393 DO pin
const int neoPin  = 6;   // WS2812B DIN
const int numLEDs = 16;

Adafruit_NeoPixel ring(numLEDs, neoPin, NEO_RGB + NEO_KHZ400);

void setup() {
  pinMode(ldrPin, INPUT);
  ring.begin();
  ring.setBrightness(255);
  for (int i = 0; i < numLEDs; i++)
    ring.setPixelColor(i, ring.Color(255, 200, 120));
  ring.show();
}

void loop() {
  int dark = digitalRead(ldrPin);
  for (int i = 0; i < numLEDs; i++) {
    ring.setPixelColor(i, dark == 0
      ? ring.Color(0, 0, 0)          // light detected — off
      : ring.Color(255, 200, 120));  // dark — warm white on
  }
  ring.show();
  delay(200);
}

Next Steps

The current version uses a digital sensor with on/off behavior. The natural next step is replacing the LM393 with a bare LDR and 10kΩ resistor voltage divider to enable smooth analog dimming — the ring gradually brightening as the room darkens rather than snapping on and off. Beyond that, adding a color temperature shift (cooler white in bright conditions, warmer amber in dim conditions) would make the lamp feel more natural and polished.
